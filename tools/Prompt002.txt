PROMPT 002  —  MaritimeCommunications_v002.html  (full regeneration spec, EN)

GOAL
----
Build a single-page application named exactly `MaritimeCommunications_v002.html`
that visualises the combinatorial explosion of NATO maritime Use Cases
(UC001..UCnnn) against Serials (S1..Sn), where — per MC 0195/12 — **each Serial
IS an Information Exchange Requirement (IER)**. The page is in NATO English
with a runtime EN ⇄ ES toggle. It must be **a single, self-contained `.html`
file** (no sibling assets folder, no CDN, no runtime fetches) and aligned with
NATO doctrine.


CORE CONCEPT — SERIALS ARE IERs
-------------------------------
An IER is the formal description of a tactical need to transfer information.
Per MC 0195/12 it answers:

  Who         — origin → destination
  What        — content type and format
  When        — latency / cadence
  Protection  — classification + integrity / authenticity posture
  Precedence  — Flash / Immediate / Priority / Routine (and edge labels like
                "Distress" or "Mixed (carries all)" for bearer-level Serials)
  Volume      — qualitative bandwidth / cadence

A Serial therefore carries:
  - a topology badge (S2H, S2S, SUB2A, …) — the WHO dimension shorthand
  - a name describing the operational purpose (NOT the equipment)
  - an `ier:{who,what,when,protection,precedence,volume}` record (EN/ES)
  - `typicalEquip / typicalFreq / typicalStanag` — the technical SOLUTION
    that satisfies the IER (the radio/SATCOM/TDL chosen)
  - `attackSurface` text and `lpiBase` (0..5) for the EW posture

A Use Case picks which IERs (Serials) are required for the mission.
Per (UC × IER) the renderer produces a key-value block where the LEFT column
is the variable name and the RIGHT column the value. See PER-USE-CASE CARD.

SELF-CONTAINMENT — SINGLE INLINED FILE
--------------------------------------
**Hard requirement: the deliverable is ONE `.html` file. No sibling assets
folder, no CDN references, nothing fetched at runtime.** Open from disk on a
fully air-gapped machine and it must render identically.

What is inlined inside the single file:

1. **Tailwind v3.4.16 runtime** (~440 KB) inside an inline `<script>…</script>`
   in `<head>`. Source: `https://cdn.tailwindcss.com/3.4.16` (Chrome
   User-Agent at build time).

2. **Lucide icons UMD** (~390 KB) inside an inline `<script>…</script>` in
   `<head>`. Source: `https://unpkg.com/lucide@latest/dist/umd/lucide.min.js`.

3. **Outfit + JetBrains Mono fonts** — fetch the Google Fonts CSS for both
   families with a Chrome User-Agent (returns woff2-subset URLs). Keep ONLY
   the `latin` and `latin-ext` blocks (drop cyrillic, greek, vietnamese, …).
   Download each referenced `.woff2`, base64-encode it, rewrite the
   `src: url(…)` of every `@font-face` to a `data:font/woff2;base64,…` URI.
   Paste the resulting CSS inside an inline `<style>` block in `<head>`.
   Final inlined font payload is ~120 KB across 4 variable-font files (one
   latin + one latin-ext per family — variable fonts share one file across
   all weights).

Pre-flight: grep for `</script>` and `</style>` inside the asset text. If
either appears, the inline wrapper would close prematurely. Tailwind 3.4.16
and Lucide UMD are both clean; if a future version is not, escape the
offending sequences.

Expected final HTML size: ~1.3-1.4 MB.

Build hint (PowerShell on Windows):
```
$tw  = [IO.File]::ReadAllText("tailwind.js")
$lu  = [IO.File]::ReadAllText("lucide.min.js")
$css = [IO.File]::ReadAllText("fonts.css")
Get-ChildItem "fonts/*.woff2" | ForEach-Object {
  $rel = "fonts/" + $_.Name
  $b64 = [Convert]::ToBase64String([IO.File]::ReadAllBytes($_.FullName))
  $css = $css.Replace($rel, "data:font/woff2;base64,$b64")
}
# Then string-replace the three asset references in the HTML template.
```

PER-USE-CASE CARD
-----------------
For every UC visible after filters:

- Header row: `uc-pill` (`UC001`..`UC010`), title, classification tag,
  environment dot (Friend / Not Friend / Mixed), criticality `n/3`, count of
  visible serials, per-UC copy button.

- Doctrinal box: gold left border, gold mini-section-label "AJP-3.1
  doctrinal hook" + the UC's `doctrine[lang]` paragraph.

- **Serial grid (variable-left / value-right layout)**: an auto-fit CSS grid
  (`repeat(auto-fit, minmax(360px, 1fr))`) that lays one block per
  applicable Serial. Each block contains:
    * Header strip: `serial-pill` (Sx) + `topo-pill` + serial name
    * 2-column `.kv-table`: LEFT column = variable name (uppercase, cyan,
      ~38% width); RIGHT column = value for that (UC × Serial / IER).

  Row order inside every block (14 rows — 6 IER + 8 context):
     1. IER · Who          — origin → destination (Serial)
     2. IER · What         — content & format (Serial spec + UC narrative
                             appended after a `·`)
     3. IER · When         — latency / cadence (Serial default + UC tempo
                             appended after a `·`)
     4. IER · Protection   — `<UC.classification> — <crypto-and-integrity
                             posture>`; mission classification overrides the
                             IER's notional default
     5. IER · Precedence   — Flash / Immediate / Priority / Routine /
                             Distress / Mixed (carries all) — rendered as
                             a small gold `.prec-tag`
     6. IER · Volume       — qualitative bandwidth / cadence (Serial)
     7. Frequencies / Equipment   — typicalEquip / typicalFreq / typicalStanag,
                                    multi-line, smaller mono on lines 2-3
     8. Time Window
     9. Environment        — coloured dot (env-Friend / env-NotFriend /
                             env-Mixed)
    10. Zones, CoI
    11. Availability       — High (24/7) / Medium (mission) / Best-effort
    12. Classification     — `.cls-tag` of UC's classification
    13. Risk (LPI/LPD)     — five-dot meter + `n/5` + label (LPI+LPD / LPI /
                             None) AND a one-sentence narrative below the
                             dots, drawn from a 6-element `RISK_NARR.{en,es}`
                             array indexed by `lpi`
    14. Attack Surface

  Rationale: the previous wide `<table>` with one row per Serial was hard to
  scan once columns multiplied; pivoting to "variable on the left, value on
  the right" inside one block per Serial (= one IER) gives a clean
  operator-readable card. Several blocks per UC sit side-by-side on wide
  screens and stack on mobile.


COMBINATORIAL GENERATOR  (`generateRow(uc, s)`)
-----------------------------------------------
Pure function returning an object consumed by both the DOM render and the
ASCII / TSV / CSV builders.

Time window:
  timeWindow = uc.tempo[lang] with these per-Serial overrides:
    S6  → "VLF/ELF rx continuous + PD burst slots"  /  Spanish equivalent
    S4  → "Burst at PD (<5 s) + acoustic CW"        /  Spanish
    S8  → "Scheduled broadcast windows"              /  Spanish
    S10 → "Boarding window (30-90 min)"              /  Spanish
    S12 in UC007 (SAR) → "Continuous · GMDSS 24/7 watch" / Spanish

IER assembly:
```
PROT_BY_CLS = {
  NU: "Plain (civilian-grade) or basic auth",
  NR: "Crypto-authenticated (NR)",
  NS: "Crypto-authenticated, integrity-critical (NS)",
  MS: "Crypto-authenticated + non-repudiation (MS)"
}
ier = {
  who:        serial.ier.who[lang],
  what:       serial.ier.what[lang]  + " · " + uc.ierBase[lang],
  when:       serial.ier.when[lang]  + " · " + uc.tempo[lang],
  protection: uc.classification + " — " + PROT_BY_CLS[uc.classification][lang],
  precedence: serial.ier.precedence,
  volume:     serial.ier.volume[lang]
}
```
Mission classification overrides the IER's notional default. Precedence
comes from the IER (Serial). What/When are concatenations: the IER's
default plus the UC's narrative / tempo.

Availability scoring:
```
baseAvail per Serial =
   S1:3, S2:2, S3:2, S4:1, S5:1, S6:2,
   S7:2, S8:3, S9:2, S10:2, S11:2, S12:3
score = min(3, round((baseAvail + uc.criticality) / 2))
3 → "High (24/7)", 2 → "Medium (mission)", 1 → "Best-effort"
```

LPI/LPD scoring + narrative:
```
lpi = serial.lpiBase
if uc.threat === "Friend"  → lpi = min(5, lpi + 1)
if uc.code   === "UC009"   → lpi = min(5, lpi + 1)
riskLabel = lpi >= 4 ? "LPI+LPD" : lpi >= 2 ? "LPI" : "None"
```
Narrative scale (must exist EN + ES, indexed 0..5):
  0 — "Open emission — high intercept risk; intelligible to any nearby SDR."
  1 — "Low resilience — short-range interception likely; minimal anti-jam."
  2 — "LPI features — intercept feasible by a capable adversary with DF."
  3 — "LPI hardened — intercept difficult; geolocation/DF still possible."
  4 — "LPI + LPD — intercept and detection both unlikely without close access."
  5 — "Stealth-grade — covert; compatible with EMCON Alpha and protected SATCOM."

Return shape (used by render + exports):
```
{
  serialCode, topo, serialName,
  freq:       <html with typicalEquip / typicalFreq / typicalStanag>,
  timeWindow,
  env, envKey,
  zones,
  ier:        {who,what,when,protection,precedence,volume},
  availability,
  classification,
  risk,       riskLabel,  riskNarrative,
  attackSurface,
  // Plain-text variants for ASCII export:
  plainFreq:  `${typicalEquip} | ${typicalFreq} | ${typicalStanag}`,
  plainRisk:  `${risk}/5 ${riskLabel} — ${riskNarrative}`,
  plainIer:   `Who: … | What: … | When: … | Protection: … | Precedence: … | Volume: …`
}
```


FILTERS
-------
Three independent inclusion sets, all active by default, controlled by
`.chip` toggles. UCs and rows are filtered by:
- `FILTERS.env` ⊆ {Friend, NotFriend, Mixed} — applied at UC level
- `FILTERS.cls` ⊆ {NU, NR, NS, MS}           — applied at UC level
- `FILTERS.serials` ⊆ {S1..S12}              — applied at row level
A UC is hidden entirely if it loses every visible serial. Counters update
on every render.

ASCII / TSV / CSV BUILDERS
--------------------------
Headers (18 columns):
  UC | Serial (IER) | Topo | IER · Who | IER · What | IER · When |
  IER · Protection | IER · Precedence | IER · Volume / Cadence |
  Frequencies / Equipment | Time Window | Environment | Zones, CoI |
  Availability | Class. | LPI/LPD | LPI/LPD narrative | Attack Surface

`buildMatrixRows()` walks USE_CASES → applicable serials (post-filter) and
produces rows where:
- "UC" column is `${uc.code} ${uc.title[lang]}`
- "Serial (IER)" column is `${s.code} ${s.name[lang]}`
- "Frequencies/Equipment" is the plain string
  `${typicalEquip} | ${typicalFreq} | ${typicalStanag}`
- "LPI/LPD" is `${n}/5 ${label}` and "LPI/LPD narrative" is the sentence
  from `RISK_NARR[lang][lpi]`
- Every cell is run through `stripHtml()` (replace `<br>` with `· `, drop
  other tags, collapse whitespace)

`buildTsv()` joins columns with TAB, rows with `\n`. Excel paste must split
straight into columns; absolutely no tabs inside cells (replace any with
spaces).

`buildGrid()` (Word-friendly ASCII grid):
- Per-column wrap caps (chars), in column order:
  `[28, 24, 6, 28, 36, 24, 24, 14, 24, 32, 22, 12, 24, 14, 6, 10, 36, 32]`
- `wrapText(text, width)`: greedy whitespace wrap; hard-break tokens longer
  than the cap. Returns `['']` for empty.
- Compute final per-column widths from the wrapped data (max line length).
- Border line: `'+' + widths.map(w => '-'.repeat(w+2)).join('+') + '+'`
- Header separator uses `=` instead of `-` for emphasis (replaces only the
  first separator after the header row).
- Every cell rendered as `'| ' + line.padEnd(width) + ' '` joined by ` | `,
  closed by ` |`. Multi-line cells emit one text row per wrapped line.
- Prepend a 4-line caption:
  ```
  NATO MARITIME COMMUNICATIONS — <hdrTitle>
  Generated <ISO 19> · lang=<lang> · rows=<n>
  Filters: env={..} cls={..} serials={..}
  <blank>
  ```

CSV: same matrix rows; values quoted with `"…"` only when they contain `,`,
`;`, `"` or newline (escape `"` as `""`). Separator `;`. Prepend UTF-8 BOM
(`﻿`). File name:
`MaritimeCommunications_v002_UCxLink_<lang>.csv`.
